# 🔺 Next.js Security Rules

> **Standard:** Security rules for Next.js 14+ (App Router and Pages Router) including Server Components, Server Actions, Route Handlers, and Middleware.
> **Sources:** Next.js Security Documentation, Vercel Security Advisories, OWASP Top 10:2025, CWE/MITRE, NVD/CVE Database, GitHub Advisory Database, Auth.js Security Advisories, Snyk Next.js Research
> **Version:** 1.0.0
> **Last updated:** March 2026
> **Scope:** Next.js 14+ App Router (Server Components, Server Actions, Route Handlers, Middleware), Pages Router (`getServerSideProps`, `getStaticProps`, API Routes), next-auth/Auth.js v4/v5, next/image, next/headers, next.config.js. React-specific rules (dangerouslySetInnerHTML, XSS, localStorage tokens) are covered in `code-security-react.md`.

---

## General Instructions

Apply these rules when writing or reviewing Next.js code. Next.js blurs the server/client boundary — the same codebase runs code on the server (Route Handlers, Server Components, Middleware) and in the browser (Client Components). This creates a distinct risk profile: **Server Actions accept arbitrary POST requests and require explicit authorization checks**; **Middleware runs before authentication in the request lifecycle, and a single header (`x-middleware-subrequest`) could bypass it entirely (CVE-2025-29927)**; **`NEXT_PUBLIC_*` environment variables are bundled into the client JavaScript bundle and visible to all users**; **`params` and `searchParams` are user-controlled strings that must be validated before use in SQL queries, file paths, or API calls**; and **`next/image` with permissive `remotePatterns` enables open image proxying and potential SSRF**. Every Route Handler and Server Action is effectively a backend API endpoint — apply the same rigor as Express or Fastify route handlers.

## 1. Middleware Auth Bypass — CVE-2025-29927

**Vulnerability:** In Next.js < 15.2.3 (and < 14.2.25 for the 14.x line), a crafted `x-middleware-subrequest` request header caused the middleware to skip execution entirely, bypassing authentication, authorization, and rate-limiting logic implemented in `middleware.ts`. Any user who set this header could access protected routes directly. This is the most critical Next.js vulnerability to date.

**References:** CVE-2025-29927 (Critical, CVSS 9.1), CWE-284, CWE-287

### Mandatory Rules

- **Update Next.js to ≥ 15.2.3 (or ≥ 14.2.25 for 14.x)** immediately — CVE-2025-29927 is trivially exploitable by any unauthenticated user.
- **Never implement authentication or authorization exclusively in `middleware.ts`** — middleware is the first line of defense, not the only one; enforce access control again in Route Handlers and Server Components.
- **Strip the `x-middleware-subrequest` header at the reverse proxy/CDN layer** (nginx, Vercel edge, Cloudflare) as a defense-in-depth measure on unpatched versions.
- **Run `npm audit` or `npx @next/codemod@latest` after upgrading** to verify no other breaking changes affect your middleware configuration.
- **Test middleware behavior** by sending requests with unexpected headers to protected routes — confirm that the middleware runs and returns 401/403 as expected.

```ts
// ❌ INSECURE — auth only in middleware; bypassed by CVE-2025-29927 on old versions
// middleware.ts
export function middleware(request: NextRequest) {
  const token = request.cookies.get('session');
  if (!token) return NextResponse.redirect(new URL('/login', request.url));
  return NextResponse.next();
}

// ✅ SECURE — defense in depth: middleware + server-side check in each handler
// middleware.ts (first layer)
export function middleware(request: NextRequest) {
  const token = request.cookies.get('session');
  if (!token) return NextResponse.redirect(new URL('/login', request.url));
  return NextResponse.next();
}

// app/dashboard/page.tsx (second layer — survives middleware bypass)
export default async function DashboardPage() {
  const session = await getServerSession(); // validates independently
  if (!session) redirect('/login');
  return <Dashboard />;
}
```

## 2. Server Actions Security — CSRF, Authorization, and Input Validation

**Vulnerability:** Server Actions (`'use server'`) are exposed as `POST` endpoints automatically generated by Next.js. They accept calls from any origin (including cross-site requests in some configurations), receive untyped `FormData` or raw arguments, and run with full server privileges. Missing authorization checks allow any authenticated user to call an admin action; missing CSRF protection allows cross-site form submissions.

**References:** CWE-352, CWE-284, CWE-20

### Mandatory Rules

- **Verify the caller's session and authorization in every Server Action** — never assume that calling a Server Action requires authentication just because the UI is behind a login wall; actions are callable via direct HTTP POST.
- **Check that the authenticated user owns the resource** before mutating it — validate `userId === resource.ownerId` to prevent IDOR.
- **Validate and parse all Server Action inputs with Zod** before processing — `FormData.get()` returns `string | File | null`; never pass raw values to database queries.
- **Next.js 14.1+ includes built-in CSRF origin checking for Server Actions** — do not disable it; ensure your `allowedOrigins` in `next.config.js` does not include untrusted domains.
- **Return generic error messages** from Server Actions — avoid leaking stack traces, database errors, or internal paths in the `Error` object returned to the client.
- **Rate-limit expensive Server Actions** (password reset, email sending, file upload) using middleware or an in-action counter backed by Redis/Upstash.

```ts
// ❌ INSECURE — no auth check; any logged-in user can delete any post
'use server';
export async function deletePost(postId: string) {
  await db.posts.delete({ where: { id: postId } });
}

// ❌ INSECURE — unvalidated FormData used in SQL
'use server';
export async function createComment(formData: FormData) {
  const body = formData.get('body'); // string | File | null
  await db.query(`INSERT INTO comments (body) VALUES ('${body}')`); // SQL injection
}

// ✅ SECURE — auth check + ownership validation + Zod validation + parameterized query
'use server';
import { z } from 'zod';
import { getServerSession } from 'next-auth';

const DeletePostSchema = z.object({ postId: z.string().uuid() });

export async function deletePost(input: unknown) {
  const session = await getServerSession();
  if (!session) throw new Error('Unauthorized');

  const { postId } = DeletePostSchema.parse(input);
  const post = await db.posts.findUnique({ where: { id: postId } });
  if (!post || post.authorId !== session.user.id) throw new Error('Forbidden');

  await db.posts.delete({ where: { id: postId } });
}

// ✅ SECURE — Zod-validated FormData; parameterized query
'use server';
const CommentSchema = z.object({ body: z.string().min(1).max(2000) });

export async function createComment(formData: FormData) {
  const session = await getServerSession();
  if (!session) throw new Error('Unauthorized');

  const { body } = CommentSchema.parse({ body: formData.get('body') });
  await db.query('INSERT INTO comments (body, user_id) VALUES ($1, $2)', [body, session.user.id]);
}
```

## 3. Route Handler Security (App Router API Endpoints)

**Vulnerability:** Route Handlers (`app/api/.../route.ts`) are the App Router equivalent of API Routes. They are full HTTP endpoints with no built-in authentication, rate limiting, or input validation. Without explicit guards, they accept requests from anyone. Unlike Server Actions, they do not have CSRF protection by default and accept all HTTP methods.

**References:** CWE-284, CWE-89, CWE-20, CWE-400

### Mandatory Rules

- **Authenticate every Route Handler** — verify the session or API token at the start of every handler; never rely on the route path being "obscure".
- **Validate request body schemas with Zod** before using any field in business logic or database queries.
- **Return appropriate HTTP status codes** — `401 Unauthorized` for missing auth, `403 Forbidden` for insufficient permissions, `422 Unprocessable Entity` for validation errors.
- **Set explicit `Content-Type: application/json`** on all JSON responses to prevent MIME-sniffing attacks.
- **Apply rate limiting** — use `next-rate-limit`, `@upstash/ratelimit`, or a middleware-level rate limiter on public-facing handlers.
- **Restrict allowed HTTP methods** — export only the methods your handler supports; an unexpected `DELETE` on a `GET`-only route should return `405 Method Not Allowed`.
- **Never return full database objects** — project only the fields the client needs; strip internal fields (`passwordHash`, `secret`, `internalNotes`).

```ts
// ❌ INSECURE — no auth; returns full user object including sensitive fields
// app/api/users/[id]/route.ts
export async function GET(request: Request, { params }: { params: { id: string } }) {
  const user = await db.users.findUnique({ where: { id: params.id } });
  return Response.json(user); // includes passwordHash, 2faSecret, etc.
}

// ✅ SECURE — auth + ownership check + field projection + Zod param validation
import { z } from 'zod';
import { getServerSession } from 'next-auth';

const ParamSchema = z.object({ id: z.string().uuid() });

export async function GET(request: Request, { params }: { params: { id: string } }) {
  const session = await getServerSession();
  if (!session) return Response.json({ error: 'Unauthorized' }, { status: 401 });

  const { id } = ParamSchema.parse(params);
  if (id !== session.user.id && session.user.role !== 'admin') {
    return Response.json({ error: 'Forbidden' }, { status: 403 });
  }

  const user = await db.users.findUnique({
    where: { id },
    select: { id: true, name: true, email: true, role: true }, // no sensitive fields
  });

  if (!user) return Response.json({ error: 'Not found' }, { status: 404 });
  return Response.json(user);
}
```

## 4. `NEXT_PUBLIC_*` Environment Variable Leaks

**Vulnerability:** Any environment variable prefixed with `NEXT_PUBLIC_` is embedded into the client-side JavaScript bundle at build time and is visible to every user. Secrets (API keys, database connection strings, private tokens) stored under `NEXT_PUBLIC_` are fully exposed in the browser's network tab and source files.

**References:** CWE-312, CWE-798

### Mandatory Rules

- **Never store secrets in `NEXT_PUBLIC_*` variables** — treat every `NEXT_PUBLIC_` value as publicly visible; it will appear in the compiled JavaScript bundle.
- **Move secret usage to Route Handlers or Server Actions** — the client calls your Next.js API, which uses the secret internally; the secret never reaches the browser.
- **Audit `next.config.js` `env` block** — values added there without the `NEXT_PUBLIC_` prefix are available server-side only; with the prefix they are bundled client-side.
- **Use environment variable linting** — add `eslint-plugin-nextjs` rules or a CI check that fails if a secret-pattern value (containing `SECRET`, `KEY`, `PRIVATE`, `PASSWORD`) is assigned to a `NEXT_PUBLIC_` variable.

```bash
# ❌ INSECURE — bundled into client JS; visible to all users
NEXT_PUBLIC_STRIPE_SECRET_KEY=sk_live_...
NEXT_PUBLIC_OPENAI_API_KEY=sk-proj-...
NEXT_PUBLIC_DATABASE_URL=postgresql://admin:password@db:5432/prod

# ✅ SECURE — server-only (no NEXT_PUBLIC_ prefix); never reaches browser
STRIPE_SECRET_KEY=sk_live_...       # used in Route Handler or Server Action
OPENAI_API_KEY=sk-proj-...          # used in Server Component or Server Action
DATABASE_URL=postgresql://...       # used by Prisma/Drizzle on server only

# ✅ SECURE — only public values go in NEXT_PUBLIC_
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_live_...  # safe: publishable key
NEXT_PUBLIC_APP_URL=https://app.example.com      # safe: public URL
NEXT_PUBLIC_ANALYTICS_ID=UA-XXXXXXXX-1          # safe: public ID
```

## 5. Security Headers via `next.config.js`

**Vulnerability:** Without explicit security headers, Next.js applications are vulnerable to clickjacking (missing `X-Frame-Options`), MIME sniffing (missing `X-Content-Type-Options`), and XSS (missing or permissive `Content-Security-Policy`). Next.js does not set security headers by default.

**References:** CWE-1021, CWE-79, CWE-693

### Mandatory Rules

- **Configure security headers in `next.config.js` using the `headers()` async function** — apply them to all routes (`source: '/(.*)'`).
- **Set `Content-Security-Policy`** with `script-src 'self' 'nonce-{nonce}'` — avoid `unsafe-inline` and `unsafe-eval`; use nonces for Next.js inline scripts.
- **Set `Strict-Transport-Security`** with `max-age=63072000; includeSubDomains; preload` for HTTPS-only deployments.
- **Set `X-Frame-Options: DENY`** or `Content-Security-Policy: frame-ancestors 'none'` to prevent clickjacking.
- **Set `X-Content-Type-Options: nosniff`** to prevent MIME-type sniffing attacks.
- **Set `Referrer-Policy: strict-origin-when-cross-origin`** to limit referrer information sent to third parties.
- **Set `Permissions-Policy`** to disable unused browser features (camera, microphone, geolocation).

```js
// next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          { key: 'X-Frame-Options', value: 'DENY' },
          { key: 'X-Content-Type-Options', value: 'nosniff' },
          { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
          {
            key: 'Strict-Transport-Security',
            value: 'max-age=63072000; includeSubDomains; preload',
          },
          {
            key: 'Permissions-Policy',
            value: 'camera=(), microphone=(), geolocation=()',
          },
          {
            key: 'Content-Security-Policy',
            // Use a nonce-based CSP for Next.js inline scripts
            value: [
              "default-src 'self'",
              "script-src 'self' 'nonce-REPLACE_WITH_NONCE'",
              "style-src 'self' 'unsafe-inline'",
              "img-src 'self' data: https:",
              "font-src 'self'",
              "connect-src 'self' https://api.example.com",
              "frame-ancestors 'none'",
            ].join('; '),
          },
        ],
      },
    ];
  },
};
module.exports = nextConfig;
```

## 6. next/image Security — `remotePatterns` and Open Image Proxy

**Vulnerability:** The `next/image` component proxies and optimizes images from remote URLs. An overly permissive `remotePatterns` configuration (e.g., `hostname: '**'`) turns the Next.js image optimization endpoint into an open proxy, allowing attackers to proxy arbitrary external content through your server (bandwidth abuse, content injection, potential SSRF to internal services).

**References:** CWE-918, CWE-400

### Mandatory Rules

- **Never use wildcard `hostname: '**'` or `hostname: '*'` in `remotePatterns`** — allowlist only the exact hostnames from which you load images.
- **Specify `protocol: 'https'`** in every `remotePatterns` entry — never allow `http://` image sources in production.
- **Specify `pathname` patterns** as narrowly as possible — `pathname: '/avatars/**'` is safer than `pathname: '/**'`.
- **Set `minimumCacheTTL`** on the image configuration to prevent cache exhaustion attacks via unique query parameters.
- **Validate image URLs** before passing them to `<Image src={userUrl}>` — apply the same `safeUrl()` validation as for `href` props (see `code-security-react.md` §2).

```js
// ❌ INSECURE — open proxy; any URL on any host is proxied through your server
// next.config.js
images: {
  remotePatterns: [{ hostname: '**' }],
},

// ❌ INSECURE — allows HTTP and any path
images: {
  remotePatterns: [{ protocol: 'http', hostname: 'cdn.example.com' }],
},

// ✅ SECURE — exact hostname; HTTPS only; scoped pathname
images: {
  remotePatterns: [
    {
      protocol: 'https',
      hostname: 'cdn.example.com',
      pathname: '/images/**',
    },
    {
      protocol: 'https',
      hostname: 'avatars.githubusercontent.com',
      pathname: '/u/**',
    },
  ],
  minimumCacheTTL: 60,
},
```

## 7. Open Redirect via `redirect()` and `next.config.js` Redirects

**Vulnerability:** Passing user-controlled values to Next.js's `redirect()` function, `permanentRedirect()`, or configuring `next.config.js` redirects with unvalidated `destination` values allows attackers to redirect users to malicious sites. This enables phishing attacks that leverage the trusted domain.

**References:** CWE-601

### Mandatory Rules

- **Validate redirect targets before calling `redirect()`** — only allow relative paths (starting with `/`) or an explicit allowlist of trusted absolute URLs.
- **Never read `searchParams.get('returnTo')` or `params.redirect` and pass it directly to `redirect()`** — always apply `safeRedirect()` validation first.
- **In `next.config.js` redirects, do not interpolate user-controlled path segments into `destination`** — use static destinations or server-validated paths.
- **Prefer `permanentRedirect()` for canonical URL changes** and `redirect()` for conditional logic — never use either with unsanitized input.

```ts
// ❌ INSECURE — open redirect: ?next=https://evil.com
// app/login/page.tsx
export default async function LoginPage({ searchParams }: { searchParams: { next?: string } }) {
  const session = await getServerSession();
  if (session) redirect(searchParams.next ?? '/dashboard'); // attacker controls destination
}

// ✅ SECURE — validate redirect target to relative paths only
function safeRedirect(url: string | undefined, fallback = '/dashboard'): string {
  if (!url || !url.startsWith('/') || url.startsWith('//')) return fallback;
  return url;
}

export default async function LoginPage({ searchParams }: { searchParams: { next?: string } }) {
  const session = await getServerSession();
  if (session) redirect(safeRedirect(searchParams.next));
}
```

## 8. next-auth / Auth.js Security

**Vulnerability:** Misconfigured next-auth (v4) or Auth.js (v5) can expose sessions, allow authentication bypass, or enable account takeover. Common misconfigurations include: missing or weak `NEXTAUTH_SECRET`/`AUTH_SECRET`, trusting all hosts with `trustHost: true`, insecure OAuth callback URLs, permissive `callbacks.signIn` that allows unverified emails, and exposing sensitive data via `callbacks.session`.

**References:** CWE-287, CWE-384, CWE-522

### Mandatory Rules

- **Set a strong `NEXTAUTH_SECRET` / `AUTH_SECRET`** (minimum 32 random bytes) — use `openssl rand -base64 32` and store it in a secrets manager, not in the codebase.
- **Never set `trustHost: true` in production without a reverse proxy that enforces the `Host` header** — `trustHost: true` makes Auth.js trust any `Host` header, enabling host header injection attacks.
- **Validate the `callbackUrl` in `signIn` callbacks** — restrict it to your own domain; a permissive `callbackUrl` enables open redirect after OAuth sign-in.
- **In `callbacks.session`, expose only the minimum user data** — never include `passwordHash`, admin flags not needed by the client, or internal IDs beyond what the UI requires.
- **Enable CSRF protection** — Auth.js v4/v5 includes built-in CSRF token validation for POST operations; do not disable it.
- **Validate `email` in `callbacks.signIn`** — check `user.emailVerified` for providers that support it; prevent sign-in with unverified emails if your app requires it.
- **Rotate `NEXTAUTH_SECRET` on a schedule** — secret rotation invalidates all existing sessions; include it in your key rotation procedure.

```ts
// ❌ INSECURE — weak secret; exposes sensitive fields; no email verification check
export const authOptions: AuthOptions = {
  secret: 'mysecret', // weak; in source code
  callbacks: {
    session({ session, user }) {
      session.user = { ...user }; // exposes all user fields including passwordHash
      return session;
    },
    signIn({ user }) {
      return true; // allows all sign-ins including unverified emails
    },
  },
};

// ✅ SECURE — strong secret from env; field projection; email verification
export const authOptions: AuthOptions = {
  secret: process.env.NEXTAUTH_SECRET, // 32+ bytes from secrets manager
  callbacks: {
    session({ session, token }) {
      // Expose only what the client needs
      session.user = {
        id: token.sub,
        name: token.name,
        email: token.email,
        role: token.role as string,
      };
      return session;
    },
    async signIn({ user, account }) {
      // Require verified email for credential providers
      if (account?.provider === 'credentials' && !user.emailVerified) {
        return false;
      }
      return true;
    },
    async redirect({ url, baseUrl }) {
      // Restrict callbackUrl to same origin
      return url.startsWith(baseUrl) ? url : baseUrl;
    },
  },
};
```

## 9. Server Component Data Leakage — Client/Server Boundary

**Vulnerability:** Data fetched in Server Components and passed as props to Client Components crosses the network serialization boundary — it appears in the RSC payload sent to the browser. Sensitive fields (password hashes, internal tokens, admin flags not needed by the UI) passed to Client Components are exposed in the browser's network tab.

**References:** CWE-312, CWE-200

### Mandatory Rules

- **Project database results to the minimum fields needed by the Client Component** — never pass a full Prisma/Drizzle model object to a Client Component.
- **Fetch sensitive data in Server Components and render it server-side** — if a secret is needed only for display logic (e.g., masked card number), render the masked version in the Server Component, not in the Client Component.
- **Create dedicated `DTO` types for data crossing the server/client boundary** — a `PublicUser` type that explicitly excludes `passwordHash`, `twoFactorSecret`, etc.
- **Never pass authentication tokens, session secrets, or private keys as props** — these should never leave the server-side rendering context.
- **Audit the RSC payload in DevTools** — inspect network requests for `.json` or RSC responses to verify no sensitive data is being serialized.

```tsx
// ❌ INSECURE — full Prisma User model (including passwordHash) sent to browser
// app/profile/page.tsx (Server Component)
async function ProfilePage({ params }: { params: { id: string } }) {
  const user = await prisma.user.findUnique({ where: { id: params.id } });
  return <ProfileClient user={user} />; // passwordHash, twoFactorSecret visible in RSC payload
}

// ✅ SECURE — explicit field selection; only public fields cross the boundary
async function ProfilePage({ params }: { params: { id: string } }) {
  const user = await prisma.user.findUnique({
    where: { id: params.id },
    select: { id: true, name: true, bio: true, avatarUrl: true }, // no sensitive fields
  });
  if (!user) notFound();
  return <ProfileClient user={user} />;
}
```

## 10. `params` and `searchParams` Injection

**Vulnerability:** In the App Router, `params` (dynamic route segments like `[id]`) and `searchParams` (URL query parameters) are user-controlled strings passed directly to Server Components, Server Actions, and Route Handlers. Using them without validation in database queries, file path operations, or internal API calls enables SQL injection, path traversal, and SSRF.

**References:** CWE-89, CWE-22, CWE-918

### Mandatory Rules

- **Always validate `params` and `searchParams` with Zod or a strict type assertion** before using them in any query, file operation, or external call.
- **Use `z.string().uuid()` for ID parameters** — this rejects non-UUID strings before they reach the database query.
- **Treat `searchParams` values as untrusted strings** — they may contain SQL metacharacters, path separators, or encoded traversal sequences.
- **Never interpolate `params.slug` or similar into SQL strings** — use parameterized queries with the validated value.
- **Validate pagination parameters** (`page`, `limit`) against numeric bounds — `limit=999999` can cause denial of service via unbounded database queries.

```tsx
// ❌ INSECURE — raw params used in SQL; searchParams used without validation
// app/products/[category]/page.tsx
async function ProductsPage({
  params,
  searchParams,
}: {
  params: { category: string };
  searchParams: { sort?: string; limit?: string };
}) {
  // SQL injection: category = "'; DROP TABLE products; --"
  const products = await db.query(
    `SELECT * FROM products WHERE category = '${params.category}' ORDER BY ${searchParams.sort} LIMIT ${searchParams.limit}`
  );
  return <ProductList products={products} />;
}

// ✅ SECURE — Zod validation on all user-controlled values
import { z } from 'zod';

const ALLOWED_SORT = ['name', 'price', 'created_at'] as const;

const ParamsSchema = z.object({
  category: z.string().regex(/^[a-z0-9-]+$/).max(50),
});
const SearchSchema = z.object({
  sort: z.enum(ALLOWED_SORT).default('name'),
  limit: z.coerce.number().int().min(1).max(100).default(20),
});

async function ProductsPage({ params, searchParams }) {
  const { category } = ParamsSchema.parse(params);
  const { sort, limit } = SearchSchema.parse(searchParams);

  const products = await db.query(
    // sort is from a hardcoded enum; category and limit are parameterized
    `SELECT id, name, price FROM products WHERE category = $1 ORDER BY ${sort} LIMIT $2`,
    [category, limit]
  );
  return <ProductList products={products} />;
}
```

## 11. SSRF via `fetch()` in Server Components and Route Handlers

**Vulnerability:** Server-side `fetch()` calls with user-controlled URLs in Server Components, Server Actions, or Route Handlers enable Server-Side Request Forgery. The Next.js server has access to internal services (databases, metadata endpoints, internal APIs) that are unreachable from the public internet. An attacker supplying `http://169.254.169.254/latest/meta-data/` retrieves cloud instance credentials.

**References:** CWE-918, CVE-2024-34351

### Mandatory Rules

- **Never pass user-supplied URLs directly to `fetch()` in Server Components or Route Handlers** — validate scheme and resolved IP before making the request.
- **Allowlist permitted external domains** when fetching user-supplied content — check `new URL(userUrl).hostname` against a hardcoded set of trusted hosts.
- **Block private, loopback, and link-local IP ranges** when the fetch target is determined at runtime — resolve the hostname and reject private addresses.
- **CVE-2024-34351 (Next.js < 14.1.1):** A crafted `Host` header in a request to a Server Action caused Next.js to SSRF to the attacker-controlled host during redirect handling — **update to ≥ 14.1.1**.
- **Set explicit timeouts** on server-side `fetch()` calls using `AbortController` — an unresponsive internal service should not block a request indefinitely.

```ts
// ❌ INSECURE — SSRF: userUrl = "http://169.254.169.254/latest/meta-data/"
// app/api/preview/route.ts
export async function GET(request: Request) {
  const url = new URL(request.url).searchParams.get('url');
  const response = await fetch(url!); // SSRF
  return new Response(await response.text());
}

// ✅ SECURE — allowlist of permitted hostnames; timeout
const ALLOWED_FETCH_HOSTS = new Set(['api.trusted-partner.com', 'cdn.example.com']);

export async function GET(request: Request) {
  const rawUrl = new URL(request.url).searchParams.get('url');
  if (!rawUrl) return Response.json({ error: 'Missing url' }, { status: 400 });

  let parsed: URL;
  try {
    parsed = new URL(rawUrl);
  } catch {
    return Response.json({ error: 'Invalid URL' }, { status: 400 });
  }

  if (parsed.protocol !== 'https:' || !ALLOWED_FETCH_HOSTS.has(parsed.hostname)) {
    return Response.json({ error: 'Forbidden URL' }, { status: 403 });
  }

  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(), 5000);
  try {
    const response = await fetch(parsed.toString(), { signal: controller.signal });
    return new Response(await response.arrayBuffer(), {
      headers: { 'Content-Type': response.headers.get('Content-Type') ?? 'application/octet-stream' },
    });
  } finally {
    clearTimeout(timeout);
  }
}
```

## 12. `getServerSideProps` and `getStaticProps` Data Exposure (Pages Router)

**Vulnerability:** Data returned from `getServerSideProps` is serialized into the `__NEXT_DATA__` inline JSON script tag on every SSR page response. If sensitive fields (API keys, internal tokens, admin data) are included in the returned props, they are exposed to every visitor who views the page source. This also applies to `getStaticProps` — the data is embedded in the static HTML and the `.json` fetch made during client-side navigation.

**References:** CWE-312, CWE-200

### Mandatory Rules

- **Return only the minimum fields needed by the page component** from `getServerSideProps`/`getStaticProps` — omit server-side secrets, database connection details, and internal fields.
- **Strip sensitive fields explicitly** before returning from `getServerSideProps` — do not rely on the component to ignore them.
- **Enforce access control inside `getServerSideProps`** — verify the session and redirect unauthenticated users to login before fetching any data.
- **Never return environment variables** (even `NEXT_PUBLIC_` ones) from `getServerSideProps` — they are already available client-side if properly prefixed.

```ts
// ❌ INSECURE — full database row including secret fields returned to client
export async function getServerSideProps(context: GetServerSidePropsContext) {
  const user = await db.users.findUnique({ where: { id: context.params!.id as string } });
  return { props: { user } }; // passwordHash, apiKey, internalFlags in __NEXT_DATA__
}

// ✅ SECURE — auth check + field projection
export async function getServerSideProps(context: GetServerSidePropsContext) {
  const session = await getServerSession(context.req, context.res, authOptions);
  if (!session) return { redirect: { destination: '/login', permanent: false } };

  const user = await db.users.findUnique({
    where: { id: context.params!.id as string },
    select: { id: true, name: true, email: true, role: true },
  });

  if (!user) return { notFound: true };
  return { props: { user } };
}
```

## 13. Cache Poisoning via ISR and `fetch()` Cache (CVE-2024-46982)

**Vulnerability:** Next.js 14's extended `fetch()` API caches responses by default. CVE-2024-46982 (Next.js ≤ 14.2.9) allowed a crafted HTTP request to poison the shared fetch cache, causing subsequent users to receive attacker-controlled cached responses. Additionally, Incremental Static Regeneration (ISR) with user-controlled `revalidatePath`/`revalidateTag` calls allows cache invalidation by unauthorized users.

**References:** CVE-2024-46982 (High, CVSS 7.5), CWE-345

### Mandatory Rules

- **Update Next.js to ≥ 14.2.10** to patch CVE-2024-46982 (shared fetch cache poisoning).
- **Authenticate calls to `revalidatePath()` and `revalidateTag()`** — these functions should only be callable from trusted server-side paths (webhook handlers, admin Server Actions), never from unauthenticated user requests.
- **Use `cache: 'no-store'` on `fetch()` calls that handle user-specific or sensitive data** — prevents inadvertent shared caching of personalized responses.
- **Add a secret token to ISR revalidation webhook endpoints** — verify `request.headers.get('x-revalidate-secret') === process.env.REVALIDATE_SECRET`.
- **Use `unstable_noStore()`** from `next/cache` in Server Components that render user-specific content to explicitly opt out of caching.

```ts
// ❌ INSECURE — unauthenticated revalidation endpoint; any user can bust the cache
// app/api/revalidate/route.ts
export async function POST(request: Request) {
  const { path } = await request.json();
  revalidatePath(path); // attacker triggers cache invalidation for any path
  return Response.json({ ok: true });
}

// ✅ SECURE — webhook secret verification before revalidating
export async function POST(request: Request) {
  const secret = request.headers.get('x-revalidate-secret');
  if (secret !== process.env.REVALIDATE_SECRET) {
    return Response.json({ error: 'Unauthorized' }, { status: 401 });
  }

  const body = z.object({ path: z.string().startsWith('/') }).parse(await request.json());
  revalidatePath(body.path);
  return Response.json({ revalidated: true });
}

// ✅ SECURE — opt out of caching for user-specific Server Component fetch
import { unstable_noStore as noStore } from 'next/cache';

async function UserDashboard() {
  noStore(); // ensures this component is never cached
  const data = await fetch('https://api.example.com/me', { cache: 'no-store' });
  // ...
}
```

## 14. Source Map Exposure in Production

**Vulnerability:** Shipping JavaScript source maps (`.js.map` files) to production exposes your application's original source code to anyone who opens the browser DevTools. This reveals business logic, API endpoints, authentication bypass conditions, internal architecture, and comments that may contain credentials or TODO security items.

**References:** CWE-540

### Mandatory Rules

- **Set `productionBrowserSourceMaps: false`** in `next.config.js` (the default) — never enable production source maps unless you use a service that uploads maps privately.
- **If you need source maps for error tracking** (Sentry, Datadog), use their build plugins to upload source maps to the service and then delete them from the build output before deployment.
- **Verify that `.map` files are not served** by your production server — add a test to your CI pipeline that checks the absence of `*.js.map` files in the deployed bundle.

```js
// ❌ INSECURE — source maps exposed in production bundle
// next.config.js
module.exports = { productionBrowserSourceMaps: true };

// ✅ SECURE — source maps disabled (this is the default; make it explicit)
module.exports = { productionBrowserSourceMaps: false };

// ✅ SECURE — upload source maps to Sentry; delete from build before deploy
// next.config.js (with @sentry/nextjs)
const { withSentryConfig } = require('@sentry/nextjs');
module.exports = withSentryConfig(nextConfig, {
  hideSourceMaps: true, // uploads to Sentry; removes from browser bundle
});
```

## 15. Middleware Security — Scope, Performance, and Secret Handling

**Vulnerability:** Middleware runs on every matched request before authentication, caching, or routing. Mistakes in middleware — like performing expensive database calls, storing secrets in the middleware context incorrectly, or matching too broadly — create performance problems, secret leakage, and security gaps. Middleware in Next.js Edge Runtime has restricted Node.js APIs and cannot use all npm packages.

**References:** CWE-284, CWE-312

### Mandatory Rules

- **Use `config.matcher` to scope middleware to protected routes only** — avoid running auth middleware on static assets (`/_next/static`, `/favicon.ico`, `/images`) to prevent unnecessary overhead.
- **Never perform database calls in Middleware** — use lightweight JWT/session cookie verification instead; the Edge Runtime has latency constraints and the database client may not be available.
- **Verify JWT signatures in Middleware using the Web Crypto API** (`jose` library) — do not store the decoded payload directly in a header without verification, as headers are user-controlled.
- **Never expose secrets via `NextResponse.headers.set()`** — headers set in middleware are visible in the browser's network tab.
- **Log security events from middleware** (auth failures, rate limit hits) to a structured logging endpoint — do not log sensitive header values.

```ts
// ❌ INSECURE — middleware queries DB (slow; may not work in Edge Runtime)
// middleware.ts
export async function middleware(request: NextRequest) {
  const sessionId = request.cookies.get('sessionId')?.value;
  const session = await db.sessions.findUnique({ where: { id: sessionId } }); // DB call in edge
  if (!session) return NextResponse.redirect(new URL('/login', request.url));
}

// ❌ INSECURE — secret header forwarded to client
export function middleware(request: NextRequest) {
  const response = NextResponse.next();
  response.headers.set('X-Internal-Secret', process.env.INTERNAL_SECRET!); // visible in DevTools
  return response;
}

// ✅ SECURE — lightweight JWT verification; scoped matcher; no DB calls
import { jwtVerify } from 'jose';

const secret = new TextEncoder().encode(process.env.NEXTAUTH_SECRET!);

export async function middleware(request: NextRequest) {
  const token = request.cookies.get('next-auth.session-token')?.value;
  if (!token) return NextResponse.redirect(new URL('/login', request.url));

  try {
    await jwtVerify(token, secret); // cryptographic verification; no DB needed
    return NextResponse.next();
  } catch {
    return NextResponse.redirect(new URL('/login', request.url));
  }
}

export const config = {
  matcher: ['/dashboard/:path*', '/admin/:path*', '/api/protected/:path*'],
};
```

## Security Checklist

### Critical Updates
- [ ] Next.js ≥ 15.2.3 or ≥ 14.2.25 (CVE-2025-29927 middleware bypass)
- [ ] Next.js ≥ 14.2.10 (CVE-2024-46982 cache poisoning)
- [ ] Next.js ≥ 14.1.1 (CVE-2024-34351 SSRF via Host header)
- [ ] next-auth ≥ 4.24.6 / Auth.js ≥ 5.0.0-beta.4 (open redirect in callbackUrl)

### Authentication and Authorization
- [ ] Auth enforced in middleware AND in each Route Handler / Server Component (defense in depth)
- [ ] Every Server Action checks session before executing
- [ ] Ownership validated in Server Actions (userId === resource.ownerId)
- [ ] `NEXTAUTH_SECRET` / `AUTH_SECRET` is 32+ random bytes from secrets manager
- [ ] `callbackUrl` restricted to same origin in next-auth redirect callback

### Server Actions
- [ ] All inputs validated with Zod before use
- [ ] Auth check at the start of every Server Action
- [ ] Generic error messages returned (no stack traces)
- [ ] Rate limiting on sensitive actions (password reset, email, file upload)

### Route Handlers
- [ ] Session / API token verified on every handler
- [ ] Request body validated with Zod
- [ ] Only necessary fields returned (no full DB objects)
- [ ] Rate limiting applied to public endpoints

### Environment Variables
- [ ] No secrets in `NEXT_PUBLIC_*` variables
- [ ] All secret keys are server-only (no `NEXT_PUBLIC_` prefix)
- [ ] `.env` files audited for accidentally exposed credentials

### Security Headers
- [ ] `X-Frame-Options: DENY` or `frame-ancestors 'none'` in CSP
- [ ] `X-Content-Type-Options: nosniff`
- [ ] `Strict-Transport-Security` with `includeSubDomains; preload`
- [ ] `Content-Security-Policy` without `unsafe-inline` / `unsafe-eval`

### next/image
- [ ] No wildcard `hostname: '**'` in `remotePatterns`
- [ ] All `remotePatterns` use `protocol: 'https'`
- [ ] `pathname` restricted to specific paths

### Redirects
- [ ] `redirect()` targets validated to relative paths only
- [ ] No user-controlled URL passed directly to `redirect()` or `permanentRedirect()`

### Data Exposure
- [ ] Server Components pass only necessary fields to Client Components
- [ ] `getServerSideProps` / `getStaticProps` return only necessary fields
- [ ] RSC payload inspected in DevTools for sensitive data

### Params and Query Validation
- [ ] All `params` and `searchParams` validated with Zod before use
- [ ] ORDER BY / column names use hardcoded allowlists
- [ ] Pagination `limit` bounded to a maximum value

### SSRF
- [ ] User-supplied URLs validated before server-side `fetch()`
- [ ] Allowlist of permitted hostnames for external fetches

### Caching and ISR
- [ ] `revalidatePath` / `revalidateTag` endpoints require secret token
- [ ] User-specific Server Components use `noStore()` / `cache: 'no-store'`
- [ ] `productionBrowserSourceMaps: false` in `next.config.js`

### Middleware
- [ ] `config.matcher` scoped to protected routes only (excludes static assets)
- [ ] No database calls in Middleware
- [ ] JWT verification uses `jose` with secret from env
- [ ] No secrets forwarded via `NextResponse.headers.set()`
